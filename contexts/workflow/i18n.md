# i18n (Internationalization) Context

> Use this context for dual-language (e.g. Hebrew/English–style) or multi-locale projects.

---

## Core Rules

**User-facing strings MUST exist in all supported locales. No hardcoded copy. Use stable, descriptive keys.**

- Every user-visible string goes through the translation layer.
- Keys are namespaced and descriptive (e.g. `common.save`, `form.errors.emailRequired`).
- Avoid keys that duplicate the English text; they are brittle and hard to refactor.

---

## Example Patterns

### React / Next.js (web)

Use `useTranslation()` and `t('namespace.key')`:

```tsx
import { useTranslation } from 'next-intl';  // or react-i18next, etc.

function ContactForm() {
  const { t } = useTranslation('contact');

  return (
    <form>
      <label>{t('contact.nameLabel')}</label>
      <input placeholder={t('contact.namePlaceholder')} />
      <button type="submit">{t('common.submit')}</button>
    </form>
  );
}
```

### React Native / Expo

Use `useTranslation()`, `i18n.language`, and `isRTL` to drive alignment (`text-right` / `text-left`):

```tsx
import { useTranslation } from 'react-i18next';

function ProfileHeader() {
  const { t, i18n } = useTranslation('profile');
  const isRTL = i18n.language?.startsWith('he') || i18n.language?.startsWith('ar');
  const textAlign = isRTL ? 'text-right' : 'text-left';

  return (
    <Text className={`text-base font-bold ${textAlign}`}>
      {t('profile.title')}
    </Text>
  );
}
```

---

## i18n × RTL Rules

- **Use language to drive alignment:** RTL locales → labels/titles right-aligned; LTR → left-aligned. Prefer logical properties (`text-start` / `text-end`) or derive alignment from `i18n.language` / `isRTL` as in the examples above.
- **Numbers, prices, phone numbers:** Always LTR. Do not reverse or RTL-wrap numerals.
- **Full RTL layout rules:** See `workflow/rtl` for flex order, scroll behavior, icons, and logical CSS.

---

## i18n Checklist

- [ ] All user-facing strings exist in every supported locale (no missing keys).
- [ ] No hardcoded user-visible text; everything goes through `t(...)` or equivalent.
- [ ] Stable, descriptive keys (namespaced, e.g. `module.screen.field`).
- [ ] Label/text alignment driven by language (RTL vs LTR), not fixed `text-left`/`text-right`.
- [ ] Numbers, prices, and phone numbers always LTR.
